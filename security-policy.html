1. Our Security Commitment

Security is foundational to everything Vortex Technologies builds. We take a security-first approach to product development and operations, implementing controls at every layer of our stack — from infrastructure to application code to organisational process.

This policy describes our security posture and practices. We publish it openly because transparency builds trust, and trust is essential to the products we build.

2. Scope

This Security Policy applies to:

  • All Vortex Technologies digital properties, including vtechs.com.jm and all product domains
  • All software products developed by Vortex Technologies (Learnaz, FinCore, Conductor, OnTrack, Home & Farm Automation, and any future products)
  • Internal systems, infrastructure, and tooling used by Vortex Technologies employees and contractors
  • Services provided to clients under Vortex Technologies contracts

Individual products may publish supplemental security documentation covering product-specific controls and configurations.

3. Infrastructure Security

Cloud & Hosting

Our products and services are hosted on reputable cloud platforms with established security certifications (e.g., ISO 27001, SOC 2). We leverage managed services where appropriate to benefit from the security controls built into the underlying platform.

Network Security

  • All traffic between clients and our services is encrypted using TLS 1.2 or higher. We enforce HTTPS across all endpoints.
  • Firewall rules and network segmentation restrict access to production systems to authorised sources only.
  • Unnecessary ports and services are disabled by default.
  • DDoS protection is applied at the network edge.

Server & System Hardening

  • Operating system and software packages are kept up to date with security patches applied promptly.
  • Default credentials are changed and unused accounts are removed at provisioning time.
  • System configurations follow security hardening baselines appropriate to the platform and service.

Monitoring & Alerting

  • Infrastructure and application logs are centrally collected and monitored for anomalous activity.
  • Automated alerts notify the engineering team of security-relevant events, including authentication failures, unusual access patterns, and resource spikes.
  • Uptime and performance monitoring is in place for all production services.

4. Application Security

Secure Development Lifecycle

Security is integrated into our development process from the outset. Key practices include:

  • Threat modelling and security review during the design phase for new features and significant changes.
  • Code review requirements before merging to production branches.
  • Automated static analysis and dependency scanning in CI/CD pipelines to catch known vulnerabilities early.
  • Security testing (including manual review for high-risk components) prior to major releases.

Common Vulnerability Mitigations

Our applications are built with defences against the OWASP Top 10 and other common vulnerability classes, including:

  • Input validation and output encoding to prevent injection attacks (SQL injection, XSS, command injection).
  • Parameterised queries and ORM usage to prevent SQL injection at the data layer.
  • CSRF protection on all state-changing requests.
  • Secure session management with appropriate token expiry and rotation.
  • Content Security Policy (CSP) and other security headers on web properties.
  • Rate limiting and brute-force protection on authentication endpoints.

Dependency Management

We monitor third-party dependencies for known vulnerabilities using automated scanning tools. Critical vulnerabilities in dependencies are remediated as a priority. We avoid unmaintained dependencies in production systems.

Authentication & Authorisation

  • Passwords are stored using a strong one-way hashing algorithm (e.g., bcrypt or Argon2). Plaintext passwords are never stored or logged.
  • Multi-factor authentication (MFA) is supported and encouraged for user accounts, and required for administrative access.
  • Authorisation checks are enforced server-side. Client-side controls are considered cosmetic only.
  • Role-based access control (RBAC) limits access to data and functionality based on user roles and permissions.

5. Data Security

Encryption

  • In transit: all data transmitted between clients and our services is encrypted using TLS. Internal service-to-service communication within our infrastructure is also encrypted.
  • At rest: sensitive data stored in databases and file storage is encrypted at rest using AES-256 or equivalent.
  • Backups: backup data is encrypted and access is restricted to authorised personnel.

Data Minimisation

We collect and retain only the personal data necessary for the purposes described in our Privacy Policy. Unnecessary data is purged according to documented retention schedules.

Backup & Recovery

Production data is backed up regularly. Backup integrity is tested periodically. Recovery procedures are documented and tested to ensure we can restore service in the event of data loss or system failure.

6. Access Control

  • Access to production systems and sensitive data is granted on a least-privilege, need-to-know basis.
  • All administrative access requires MFA.
  • Access rights are reviewed periodically and revoked promptly when no longer required (e.g., upon role change or departure).
  • Privileged access is logged and auditable.
  • SSH key-based authentication is required for server access; password-based SSH authentication is disabled.
  • Secrets (API keys, credentials, tokens) are stored in a secrets management system and are not committed to source code repositories.

7. Incident Response

We maintain a documented incident response process to handle security events efficiently and minimise impact. Key elements include:

  • Detection: monitoring and alerting systems are designed to detect security events promptly.
  • Triage & containment: confirmed incidents are assessed for severity and contained to prevent further impact.
  • Investigation & remediation: root cause analysis is conducted, and the underlying issue is remediated before affected systems are restored to production.
  • Notification: affected users and, where required, regulatory authorities will be notified in accordance with applicable law. We aim to notify affected users within 72 hours of confirming a breach involving their personal data.
  • Post-incident review: a lessons-learned review is conducted after significant incidents to improve our controls and processes.

To report a suspected security incident, contact info@vtechs.com.jm.

8. Vendor & Third-Party Management

We assess the security posture of third-party vendors and service providers before engaging them. Our evaluation considers:

  • Security certifications and compliance status (e.g., ISO 27001, SOC 2 Type II)
  • Data processing agreements and contractual security obligations
  • Data localisation and transfer mechanisms where personal data is involved
  • Vendor incident response and notification commitments

We limit the personal data shared with third parties to the minimum necessary, and we conduct periodic reviews of active vendor relationships.

9. Compliance & Standards

Our security programme is informed by recognised frameworks and standards, including:

  • OWASP Top 10 — for application security guidance
  • NIST Cybersecurity Framework — for overall programme structure (Identify, Protect, Detect, Respond, Recover)
  • CIS Controls — for infrastructure hardening baselines
  • Jamaica Data Protection Act 2020 — for personal data handling obligations

As our products grow in scope and user base, we are committed to achieving and maintaining relevant certifications and compliance status.

10. Responsible Disclosure

We welcome reports from security researchers who identify potential vulnerabilities in our products and infrastructure. We are committed to working with the security community in a cooperative and transparent manner.

How to Report a Vulnerability

Email your findings to info@vtechs.com.jm. Please include:

• A clear description of the vulnerability and its potential impact
• The affected product, system, or URL
• Steps to reproduce, including any proof-of-concept (where safe to share)
• Your contact information for follow-up

We will acknowledge your report within 3 business days and keep you informed of our progress. We aim to resolve confirmed vulnerabilities within 90 days, depending on severity and complexity.

Our Commitments to Researchers

  • We will not pursue legal action against researchers who follow this responsible disclosure process in good faith.
  • We will keep your report confidential and will not share your identity without your consent.
  • We will credit you publicly for your finding (if you wish) once the issue is resolved.

Out-of-Scope Activities

The following are outside the scope of our responsible disclosure programme and should not be performed:

  • Denial-of-service (DoS/DDoS) testing
  • Automated scanning that generates excessive load on production systems
  • Social engineering or phishing of Vortex Technologies employees
  • Physical security attacks
  • Accessing, modifying, or deleting data belonging to other users
  • Testing third-party services or infrastructure not controlled by Vortex Technologies
